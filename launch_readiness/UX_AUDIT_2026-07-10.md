# Frontend UX & Accessibility Audit — 2026-07-10

Read-only design/accessibility audit of the **frontend** (React Native / Expo SDK 53, TypeScript,
NativeWind, Expo Router), reviewed as a product designer specializing in older adults (target
audience 40+: presbyopia and reduced fine-motor precision are common). Complements `BACKLOG.md`
and the design work in `../design_handoff_ux_remediation/` (MF- tickets); overlaps are
cross-referenced.

**Priorities** (same scale as `BACKLOG.md`): **P0** = go/no-go blocker for store submission ·
**P1** = should land before launch · **P2** = real gap, can fast-follow · **P3** = nice-to-have.

**Method:** source-based audit — font sizes, touch-target dimensions, and contrast ratios are
literal values in the NativeWind classes and `lib/theme.ts` tokens, so this is more precise than
screenshots for those questions. Contrast ratios below were computed from the actual token hex
values against the WCAG formula. **Not covered (needs a device pass):** real Dynamic-Type layout
behavior at max font size, actual on-device tap accuracy, and VoiceOver/TalkBack flow. Evidence
file:line references are as of 2026-07-10 — re-verify before acting.

**One-line summary:** touch targets, loading/empty states, and the waiver are genuinely strong;
the accessibility work (MF-007…010) was real but stopped at the **default theme and the happy
path**. The findings that most hurt a 40+ first-timer are all off the happy path (onboarding data
loss, empty first-value moment, silent error screens), plus the alternate color themes ship
sub-AA text.

---

## CRITICAL (P0)

- [ ] **UX-01 · P0** — Onboarding answers are never persisted; backgrounding the app wipes all 7
      steps. `components/onboarding-form.tsx:82-101` holds the questionnaire in `useState` only —
      no `AsyncStorage`/`SecureStore` write, no draft restore. A user who gets a phone call
      mid-onboarding and has the app evicted restarts from step 1. Direct abandonment driver for a
      low-confidence audience. **Fix:** persist `formData` on each step advance, restore on mount.
      **Effort: M.**

- [ ] **UX-02 · P0** — First-value moment dumps the user on an empty analytics dashboard while
      their plan generates silently. `components/onboarding/use-onboarding-controller.ts:71-108`
      fires `startWorkoutGeneration()` fire-and-forget, then `router.replace("/(tabs)/dashboard")`
      — a screen of empty weight/strength charts, no "building your plan" moment, no landing on the
      actual workout. **Fix:** route to a dedicated "Building your first week — about a minute"
      screen (reuse `WarmingUpScreen` / `workout-generation-modal.tsx`), then land the user **on the
      generated workout**, not the analytics tab. **Effort: M.**

- [ ] **UX-03 · P0** — Core screens fail silently on network/API errors (blank screen, no message,
      no retry). The data layer tracks `error` (`hooks/use-app-data.ts:116,814`) and the context
      re-exposes it, but no screen reads it: `dashboard-screen.tsx:144`, `calendar-screen.tsx:64`
      (and `:123` hardcodes `const error = null`, making its written-but-dead retry UI unreachable),
      `search-view.tsx:58`, `settings-view.tsx:53` all destructure only `{data, loading}`. A 40+
      user reads a blank/stale screen as "broken app." **Fix:** one shared `<DataErrorState
      onRetry/>` (mirror `NoActiveWorkoutCard`) rendered when `error && !loading`; the plumbing
      already exists — closes all four screens at once. **Effort: M.**

## HIGH

- [ ] **UX-04 · P1** — First-run generation failure is invisible.
      `use-onboarding-controller.ts:53-68,95` only `logger.error`s (stale
      `// Consider surfacing a user-friendly message` comment), then navigates to the dashboard
      anyway. A new user whose plan fails is stranded on empty charts. The dashboard's own
      `handleGenerateNewWorkout` already shows error dialogs — just not wired to the first-run
      path. **Fix:** surface "We couldn't build your plan — tap to retry." **Effort: S.**

- [ ] **UX-05 · P1** — The four alternate color themes fail WCAG AA contrast; MF-007's fixes only
      touched the default theme. Computed from `lib/theme.ts`:
      `text.muted` on background is **< 4.5:1 (AA body fail) in 8 of 10 theme variants**, and on
      cards (`--color-card` = `neutral.light[2]`) it fails in **all 10**; `warning` amber on light
      backgrounds is **~2.1–2.3:1 — fails even large-text AA** in every alternate light theme
      (safety-relevant); `danger` is borderline (3.5–3.9) in the light alternates;
      carbon-violet-dark `muted`-on-card is 2.91. Sample ratios:

      | Theme | muted/bg | muted/card | warning/bg |
      |---|---|---|---|
      | original-light (default) | 4.61 ✓ | 4.19 ✗ | 3.19 (large only) |
      | steel-blue-light | 4.06 ✗ | 3.84 ✗ | 2.16 ✗ (fails large) |
      | dusty-denim-light | 3.85 ✗ | 3.74 ✗ | 2.15 ✗ |
      | dusty-sage-light | 3.96 ✗ | 3.80 ✗ | 2.26 ✗ |
      | carbon-violet-light | 4.43 ✗ | 4.09 ✗ | 2.24 ✗ |
      | carbon-violet-dark | 3.52 ✗ | 2.91 ✗ | — |

      A 40+ user who picks any non-default theme gets *worse* legibility than the default. **Fix:**
      re-tune `muted`/`warning`/`danger` per palette to ≥4.5:1 body; the default theme is the
      template. (Extends MF-007 to the alternate palettes.) **Effort: M.**

- [ ] **UX-06 · P1** — Settings is hidden behind an unlabeled person icon, and disappears entirely
      on the Workout tab. `components/header.tsx:130-134` puts Edit Profile, Personal Info, Logout,
      Delete Account, Subscription, Legal, App Settings behind a single unlabeled `person` glyph
      (reads as "profile," not "menu"). `header.tsx:63-66` returns `null` on the workout route, so
      from the Workout tab there's **no way to reach Settings or Search**. **Fix:** label it / use a
      gear or hamburger (or promote Settings to a 4th labeled tab); keep a minimal header on the
      idle Workout screen. **Effort: S.**

- [ ] **UX-07 · P1** — The "Workout In Progress" modal surfaces a destructive red "Abandon Workout"
      from an ordinary tab tap. `app/(tabs)/_layout.tsx:42-123` disables Dashboard/Calendar mid-
      workout and pops a dialog with "Abandon Workout" one tap away, whose safe option reads
      "Continue Workout" (ambiguous — means "stay here"). A user peeking at metrics gets a data-loss
      scare. **Fix:** don't surface Abandon from a nav tap; reword the safe path to "Stay in
      Workout" as primary, note progress auto-saves; consider read-only dashboard access. **Effort: S.**

- [ ] **UX-08 · P1** — No "Step X of 7" progress text and no first-run coaching. The onboarding
      indicator (`onboarding-header.tsx`, `progressive-indicator.tsx`) is bare colored bars with no
      numbers; there is no first-run tour anywhere for someone who's never used a workout app.
      **Fix:** add "Step 3 of 7" text; add a one-time 2–3 step coach overlay on first
      workout/dashboard view. **Effort: S/M.**

## MEDIUM

- [ ] **UX-09 · P2** — Weak typography hierarchy + unused type scale. The shared `Text` component
      (`components/text.tsx:42-53`) renders `h1`/`h2`/`h3`/`title` **all at `text-xl` / 20px** — no
      heading distinction, largest heading only 20px; the `2xl/3xl/4xl` tokens (24/28/32) are
      defined but never used by the primitive. Compounding it, the shared `Text` component is used
      in only **4 files while ~78 use raw `RNText`** with ad-hoc `text-*` classes, so hierarchy is
      inconsistent app-wide. **Fix:** give headings a real scale (h1 `3xl`, h2 `2xl`, h3 `xl`) and
      route more screens through `Text`. **Effort: M.**
- [ ] **UX-10 · P2** — Dark-mode switch colors hardcoded (**this is BUGS.md BUG-004**).
      `components/profile-override-form.tsx:637-638,657-658` hardcodes `trackColor`/`thumbColor`
      (`#E5E7EB`/`#9CA3AF`/`#FFFFFF`) instead of theme tokens. Other token leaks: `text.tsx`'s
      `components.card.base` hardcodes `bg-white` (bypasses MF-010 card token),
      `circuit-timer.tsx:533` uses raw `text-red-600`/`text-blue-600`, `input.error` uses
      `border-red-500`. **Fix:** route through `danger`/`card`/theme tokens. **Effort: S.**
- [ ] **UX-11 · P2** — "Welcome back" shown to brand-new users. `components/login/login-screen.tsx:283`
      titles the first screen "Welcome back" until the name field appears after the first Continue
      tap. **Fix:** neutral title, or check email on blur. **Effort: S.**
- [ ] **UX-12 · P2** — "Regenerate Today" (destructive) sits equal-weight beside "Edit Workout."
      `components/calendar/sections/action-buttons.tsx:67-120` — regenerate discards the current AI
      workout but looks identical to edit, with a mismatched gear icon. **Fix:** de-emphasize to
      secondary/outline, use the `refresh` icon, confirm before discarding. **Effort: S.**
- [ ] **UX-13 · P2** — Android hardware-back unhandled on waiver and onboarding. No `BackHandler`
      anywhere (grep-confirmed); on Android, back on the onboarding stepper can exit the flow and
      lose progress. **Fix:** `BackHandler` that steps back (not exits) in onboarding; block/confirm
      on the waiver. **Effort: S.**
- [ ] **UX-14 · P2** — Onboarding friction cluster: OTP resend locked 54s with no "check spam" hint
      (`verify-screen.tsx:537-556`); Health-connect step has no in-step "Skip"
      (`health-connect-step.tsx`); Style step assumes jargon ("HIIT", "Plyo Box", "Rehab" —
      `workout-style-step.tsx:37-106`). **Fix:** spam hint + ~15s resend; explicit "Skip for now";
      plain-language labels first. **Effort: S.**
- [ ] **UX-15 · P2** — Hardcoded "Hello Rich!" ships on the AI-provider screen.
      `app/ai-provider-selection.tsx:309` renders a literal string instead of `user?.name`. Dev-
      gated, low blast radius, but shipped. **Fix:** use `user?.name`. **Effort: S.**
- [ ] **UX-16 · P2** — Dynamic Type on but untested at large sizes. `allowFontScaling` defaults true
      (good), but three spots cap it at `maxFontSizeMultiplier={1.3}`
      (`calendar/sections/action-buttons.tsx`, `workout-regeneration-modal.tsx`), and many fixed-
      height containers (`min-h-[44px]`, fixed cards) risk clipping at max accessibility sizes.
      **Fix:** device-test at largest Dynamic Type; relax the 1.3 caps where layout allows.
      **Effort: S (test) / M (fixes).**

- [ ] **UX-18 · P3** — `danger` body text borderline (3.5–3.9:1) in light alternate themes —

      Refactor into `components/workout/`.

## Confirmed good (preserve)

- **Touch targets** — model implementation: `Button` enforces `min-h-[44px]`, `IconButton` is a
  fixed 44×44 + `hitSlop`, reusable `HIT_SLOP` presets (`constants/accessibility.ts`, MF-009).
- **Body-text floor** raised to 16px for the audience (MF-008).
- **Loading states** — purpose-built skeletons everywhere, never a bare spinner.
- **Empty states** — shared `NoActiveWorkoutCard` grammar across dashboard/workout/calendar.
- **Workout-generation modal** — best-in-class (per-day timeline, rotating narration, reduced-motion).
- **Waiver** — plain-English, readable, not a wall of text.
- **Medical/limitations step** — tap-tiles over free text with empathetic "why we ask" framing.
- **OTP inputs** — large 62×70px cells, autofill, paste, shake-on-wrong.
- **Token system** — disciplined (CSS variables, named tonal steps, MF-tagged rationale). UX-05 is
  bad *values* in 4 palettes, not bad architecture.
