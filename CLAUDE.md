# MastersFit Frontend — Claude Code Guide

This is the **mobile app** for MastersFit, a fitness-tracking app. It runs on iOS and
Android, built with **React Native + Expo**. It talks to the API server
(`masters-fit-backend`) over HTTP.

> **New here?** Read this whole file once — it's the "house rules." When you ask Claude Code
> to add a screen, build a component, or ship to iOS, it will load a matching **skill** (a
> step-by-step recipe) automatically. You don't need to memorize the steps.

---

## What this app is

- **React Native + Expo** with **Expo Router** for navigation (file-based routing in `app/`).
- Styled with **NativeWind** (Tailwind CSS classes in React Native).
- Subscriptions via **RevenueCat**; analytics via **Mixpanel**; errors via **Sentry**.
- Health data via Apple Health / Health Connect integrations.

## How to run it

```bash
npm install              # first time only
npm start                # starts Expo (Metro bundler), then press i (iOS) or a (Android)
```

The app expects the backend at `http://localhost:5000/api` (iOS simulator) — so **start the
backend first** (`npm run dev` in `masters-fit-backend`). On Android, the dev API URL is a LAN
IP set in `config.ts` (`getApiUrl()`); update it to your machine's IP if the app can't reach
the server.

You need a `.env` file (ask the team — never commit it). See `.env.example` for the keys.

## Architecture map — where things live

The single most important rule in this codebase:

> **`app/` is for navigation only. All UI and logic live in `components/`.**

A route file in `app/` should be tiny — it just renders a component. Example
(`app/(tabs)/calendar.tsx`):

```tsx
import { CalendarScreen as CalendarComponent } from "@/components/calendar";
export default function Calendar() {
  return <CalendarComponent />;
}
```

| Folder | Job |
|--------|-----|
| `app/` | **Routing only.** Each file is a screen/route (Expo Router). Renders a component, nothing more. `(auth)` and `(tabs)` are route groups. |
| `components/` | **All UI + component-level logic.** Simple components are one file (`button.tsx`); complex screens are a folder with `index.ts` (barrel) + `<name>-screen.tsx` + `sections/`. |
| `hooks/` | Reusable stateful logic, named `use-*.ts` (e.g. `use-workout-session.ts`). |
| `contexts/` | React context providers (auth, theme, etc.). |
| `lib/` | The data layer. `lib/api.ts` is the HTTP client (`apiRequest`); per-domain files (`workouts.ts`, `profile.ts`, …) wrap it. Also analytics, logger, theme. |
| `types/` | Shared TypeScript types (`@/types/api`, etc.). |
| `constants/` | Static config and constants. |
| `utils/` | Small pure helpers. |

**Never fetch directly with `fetch()` in a component.** Go through the `lib/` layer — it
handles auth tokens, token refresh, and the paywall/waiver redirects centrally.

## Conventions (match the existing code)

From the team's naming rules (`.trae/rules/project_rules.md`):

- **File names: kebab-case** → `use-onboarding-controller.ts`, `workout-card.tsx`.
- **Components: PascalCase** → `function WorkoutCard()`.
- **Functions/variables: camelCase** → `startWorkoutGeneration`, `isGeneratingWorkout`.
- **Hooks/controllers** that hold logic are prefixed `use` → `useOnboardingController`.
- **Constants: UPPER_SNAKE_CASE** → `REGENERATION_TYPE_WEEKLY`.
- **Enums: PascalCase** → `RegenerationType`.

Other conventions:
- **Import aliases** (from `tsconfig.json`): `@/*` (repo root), plus `@components/*`,
  `@hooks/*`, `@lib/*`, `@contexts/*`, `@utils/*`, `@types/*`, `@assets/*`. Use them instead of
  long relative paths.
- **Styling is NativeWind** — use Tailwind `className="..."` strings. For theme-aware colors,
  use the theme helpers in `lib/theme` (e.g. `useThemeColors`) rather than hardcoding hex.

## 🚧 Guardrails — do not skip these

1. **Don't put logic in `app/`.** If you're writing real UI or state in an `app/` file,
   move it into a component and have the route render it.
2. **Don't call `fetch()` from components.** Use `lib/api.ts` / the domain files in `lib/`.
3. **Never commit `.env`** or secrets. Native build folders `ios/` and `android/` are mostly
   generated by `expo prebuild` — avoid hand-editing them unless you know why.
4. **Before saying a change is "done", run lint and confirm it passes:**
   ```bash
   npm run lint
   ```
   Then actually run the app (`npm start`) and look at the screen you changed. Type-checking
   and "it renders on a device/simulator" are both part of done. Report what you saw.
